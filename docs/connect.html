be10x · connect your machine

Run the agent on your own computer

The board is hosted once and holds all the state. The Claude agent runs on each teammate's own machine — their repos, their claude login — linked to the board over HTTPS. Nothing runs on the server. This is the whole flow: what you host, what a new member installs, and what's built.

01

The shape

The board lives on a host (Render, a small VM, Fly…). Teammates use it in a browser to review and steer. But the agent runs where the repo and the CLI are — on each member's own machine, linked to the board with a personal token.

02

What a new member does to connect

Four steps, once. After this a bare be10x connect just works — the setup is remembered in ~/.be10x/connect.json.

1

Get the CLI & your own Claude login

Install the be10x CLI on your machine and sign in to Claude Code as you normally would. The agent uses your login — the board never sees it.

# clone the be10x repo, then npm install
2

Mint a token on the board

Open the hosted board in your browser → Connect your machine → Create token. Copy it once — it's your machine's key to the board.

The dialog shows the exact command below with your board URL and token already filled in.

3

Link this machine & serve your repos

Point --repos at the checkouts you want to work here. This registers them with the board, writes each a board-pointing MCP config, and starts the loop.

$ be10x connect \ --board https://your-board.notpritam.in \ --token gfa_xxxxxxxxxxxx \ --repos ~/code/app,~/code/api
4

Create a task — your machine picks it up

On the board, make a task for one of those repos and hand it to the agent. Your machine claims it, runs Claude locally, and streams the plan and progress back for the team to review.

That's the whole link: a token + the be10x connect daemon. No server-side CLI, no shared keys, no repo ever uploaded.
03

What happens on each task

Every task step is the same short loop. State steps happen on the board; machine steps happen on yours. The board keeps every durability decision — so a dropped laptop or a network blip just retries.

BoardYou create a taskfor a repo you serve, and hand it to the agent
→
Your machineClaims the wakethe connector asks the board for the next job
→
Your machineClaude runs locallyin a fresh worktree, using your login
→
BoardPlan & progress stream backvia the gfa_* tools over HTTPS — team reviews
→
BoardReport & durabilityretry on failure, hand-off to verify on success

The agent process is disposable — it can die and be re-run from the board's saved state. That's the one rule the whole system keeps: sessions disposable, state durable.

04

Three roles, and only one of them is a one-time setup.

05

Status — what's built

The distributed flow above is built and tested end-to-end (228 tests). It's additive — single-host be10x serve is unchanged.

✓ Built & live

  • ✓Agent gateway — /api/agent/rpc exposes every gfa_* tool over HTTPS (token-auth), reusing the same registry as the local MCP.
  • ✓Runner API — claim · report · projects; the board keeps all retry/verify durability.
  • ✓The connector — be10x connect loop + saved config + board-pointing MCP.
  • ✓Local executor — spawns your own claude in your worktree, reports the outcome.
  • ✓HTTP MCP transport — the agent's tools reach a remote board.
  • ✓Dashboard — "Connect your machine" mints a token and shows the ready-to-run command.

◇ Bring-your-own / next

  • ◇Managed Claude login — each member brings their own for now (this is by design in V1).
  • ◇Team-scoped tokens — today any valid token can drive any task; fine for a trusted team, tightening is a small follow-up.
  • ◇Personal / chat tasks remotely — remote claim serves repo tasks; project-less tasks are handled on the host.
To go live: host the board with a persistent disk, share the URL, and have each teammate run be10x connect. See the README's “Connect your machine” section for the hosting + deploy steps.
