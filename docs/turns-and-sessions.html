Is it a new session every message — or the same one, resumed?

Every step spawns a brand-new claude process. That sounds like "starts over," but it doesn't — the conversation is resumed, not restarted. Here's exactly how that works, and precisely where it stops being enough.

Durable — lives in the DB / Claude's session store Ephemeral — one OS process, dies when the step ends Not available in Claude Code itself — confirmed via official docs

02

Where the conversation actually lives

Step 4 above is a fresh OS process every single time — different pid, starts cold, no memory of its own. What carries the conversation forward is one value: sessionId. It's captured once, stored in our DB, and handed back to Claude Code on every later run so it can restore its own session.

What this is not: a single process kept running in the background, listening for more input. Every box above dies. What's persistent is the pointer, not the process — which is also why a crashed machine loses zero conversation history.
03

The hard boundary — straight from Claude Code's own docs

This part isn't a choice this codebase made. It's confirmed directly against Claude Code's official CLI reference: there is no flag, mode, or supported pattern for keeping one process open and feeding it multiple turns.

✓ What Claude Code supports

  • One prompt in, one full turn out, then the process exits — this is -p / print mode.
  • --output-format stream-json — real-time streaming of that one turn as it's generated.
  • --resume <sessionId> on the next process — full conversation history restored, cold start, different pid.

✕ What it doesn't expose

  • No --input-format / streaming-input mode — nothing keeps stdin open across turns.
  • No way to write a second message to a process that's still mid-turn.
  • No live "steer it while it's working" primitive — not in the CLI, not in the Agent SDK either.
So, for headless mode specifically: "type into it while it's actively working" isn't available through the transport this app uses today — no process driven by -p + piped stdio can do this, confirmed against code.claude.com/docs. That's not the whole story, though — see §07, where this gets tested for real.

05

Three gaps a session-based model would have to solve — confirmed absent today

Not opinions — grepped and read line by line. These are exactly the problems "make it session-based" would inherit if nothing else changes.

✕ Confirmed missing

  • No way to stop a run once it starts. Not a route in server.js, not a button in the web app, not an MCP tool — zero matches across all three. The only way a run ends is by finishing on its own, or a person manually killing the OS process outside this app entirely.
  • "Crash-safe recovery" only cleans up bookkeeping. When a runner dies mid-step, the sweep (runner.js:221-235) marks that one runs row failed so the UI stops showing a dead process as alive — it never touches wake_queue and never resumes the task. A human still has to act again to get it moving.
  • No concurrency guard per task. Two wakes for the same task can be claimed by two different runner processes at the same moment, each spawning its own agent into the same git worktree. monitor-task.mjs only notices this after the fact and logs a warning — nothing in the code prevents it.
06

This exact question was already raised once

Worth knowing before overriding it: the current shape isn't an oversight nobody examined. It's two separate, already-recorded decisions — one closed, one still open.

✕ Already rejected — always-live sessions

Orchestration v2 design doc, stated core principle:

"event-woken, ephemeral, resumable — NOT always-live … 'staying on a task' is re-waking from durable state, not a process held open."

Mirrored today as a comment at runner.js:108-111.

✓ Already planned — stop/abort a run

Same design doc's data model sketched:

"register pid/pgid for kill; graceful CancellationToken-then-SIGKILL of the whole group" — plus a runs.status of 'killed'.

Never built: schema.sql today has only starting/running/done/failed and a bare pid, no pgid.

So: extending the one existing auto-chain, and building the abort control, both continue intent that's already on record. Making sessions always-live means consciously reversing a principle the team already wrote down and reasoned through — not filling a gap nobody thought about.

Tested it live — a real pty is the missing piece, and it works

Not inferred this time — actually run, three times, in this environment. A trivial session was created (claude -p "Reply with exactly: OK"), then resumed by sessionId two different ways.

✕ Plain pipe — today's transport

claude --resume <id> with stdin from /dev/null — exactly what child_process.spawn gives you today:

Error: No deferred tool marker found in the resumed session … Provide a prompt to continue the conversation.

Exits immediately. No UI, no prompt, no way to type into it.

✓ Real pty — identical command, same session id

Same call, wrapped so the child gets an actual pty instead of a pipe:

Full interactive UI renders. The resumed transcript shows the exact prior exchange — "Reply with exactly: OK" → "OK" — restored correctly, then sits live at a ❯ prompt waiting for input.

This is what node-pty would give a WebSocket-backed browser terminal.

What this settles: the "not available" verdict in §03 was specific to headless (-p) mode's plain-pipe transport — confirmed true, that path really doesn't exist. But genuine interactive mode, under a real pty, does exactly what the original idea needed: live rendering, and a session resumed with full history intact.
