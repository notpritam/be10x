Plan Overview · v1 scope

Git for Agents One shared board where humans and agents plan, review, approve, and ship work — code or not. Grown from the chat‑v2 issue tracker.

The seed you built (the chat‑v2 forensics board) already has the two things that matter: a rich card contract and a two‑way human↔agent loop. “Git for Agents” keeps that spine and adds the missing verbs: typed tasks (not only code), a plan‑review gate before work starts, teams, an MCP interface, and self‑rating. Later phases add remote/phone access and deterministic Playwright verification.

Seed: chat‑v2‑issues (13 cards) Now building: v1 Interfaces: Web board + MCP Model: Node · SQLite · buildless UI

The universal spine

Propose → review → approve → work → verify

Every task, regardless of type, runs the same lifecycle. This is your “bounce an idea, someone reviews it or just approves, then the agent works the flow.” The review gate is the new, load‑bearing piece.

01CaptureHuman or agent adds a task. Type chosen (code‑issue, research, idea…).
→
02PlanAgent drafts the plan for that task type. Nothing is executed yet.
→
03Review (tag)Tag a teammate / yourself. They approve or request changes.
→
04Work the flowOn approve, agent runs the type’s steps. Live status on the card.
→
05Self‑rateAgent scores the result against the task’s definition‑of‑done.
→
06RecordWhat shipped/produced is written back to the card (refs, output, evidence).
Replan / retry‑N loop back to step 02 at any time — a rejected review, a failed self‑rating, or a human “try again” re‑opens the plan without losing history.
⏸ Pause for input — mid‑flow the agent can stop and ask you a scoped question (quick‑choice chips + a custom‑input box) without abandoning the task; your answer resumes it. Shown below.

Lifecycle & the worker

The board is the state machine — a worker drives it

Tasks move through explicit states. You (or the agent) mark a task ready‑to‑work; a background cron worker notices, claims it, moves it to in‑progress, and runs the flow — nobody babysits a terminal.

Backlog→ Researching→ Plan review→ Ready to work→ In progress+ changes watcher→ Verify & self‑rate→ Done
Plan review branches before it’s ready: reiterate the plan · re‑research · verify & comment.
In progress can raise needs‑input (pause for you) then resume — or bounce back to research/plan on failure.

⏱️ Cron worker

A scheduled process polls the server on an interval, finds tasks flagged ready‑to‑work, claims them, and starts the flow. Runs locally in v1 (Claude Code scheduled‑wakeup / cron); becomes the cloud runner in v2. Pickup latency = the poll interval.

🔀 Changes watcher

While a task is in‑progress, a background watcher tracks what changed — files touched, diffs, status — and streams it onto the card, so the board shows real progress, not just a spinner.

Human‑in‑the‑loop

When the agent needs you, it asks — on the card

Work shouldn’t silently stall. When a task hits a decision only you can make, the agent pauses (state → needs‑input) and posts a request into the detail sidebar. Answer with a quick choice or custom input and it resumes exactly where it left off.

The pivot — not code‑only

Typed tasks: every type is a small plugin

A task’s type defines three things — its content contract, its flow, and its definition‑of‑done. The chat‑v2 forensics contract becomes the first type. Analysts, PMs, and researchers get their own. v1 ships two types to prove it’s generic; more are just data.

💡 idea / task new

Contractproposal · rationale · acceptance criteria
Flowdraft → discuss/bounce → decide
Doneapproved & criteria met (agent‑ or human‑executed)
Team bias = team‑level conventions injected into every flow (the shared “how we work” — the team’s CLAUDE.md). It’s what makes two teams run the same task type differently.

Where a task lives

Three scopes

Personal

Your private tasks. Work solo, no review needed unless you ask for one.

Project

A repo or a workstream. Tasks keyed by a stable project id, survive moves/reinstalls.

Team

Shared board. Members join, see tasks, tag each other to review, approve, and hand off to the agent.

v1 architecture

Two front doors, one core, one store

Humans drive the web board; agents drive MCP. Both hit the same core and the same datastore, so a task an agent creates and a task you drag are the same row.

Front doors
🖥️ Web board
Buildless HTML/JS. Kanban + detail panel + review UI. Session‑cookie login.
🔌 MCP server
create · research · plan · request_input · submit_for_review · mark_ready · claim · update · rate. Auth via personal access token.
Core (shared)
⚙️ Task engine
Typed‑task model, lifecycle state machine, review gate, input requests, self‑rating.
🔐 Auth & teams
Accounts, sessions, tokens, roles, per‑team authorization.
🤖 Agent loop
Cron‑polls ready‑to‑work tasks, runs the type’s flow, watches changes, posts live status.
Store
🗄️ SQLite
users · teams · memberships · tasks · reviews · events. Single file, local‑first, hostable, back‑up‑able. Postgres later if hosted at scale.
Runtime: Node DB: SQLite Server: Fastify/Hono Auth: cookie sessions + PATs UI: buildless HTML/JS Agents: MCP

What v1 includes — “what to include initially”

In v1 vs. deferred

✓ In v1

  • Generic typed‑task engine + 2 types (code‑issue, idea/research)
  • Task lifecycle: add your own · replan · retry‑N · agent self‑rating
  • Plan‑review gate: tag a reviewer → approve / request changes → agent works
  • Personal / project / team scopes
  • Explicit lifecycle: research → plan‑review (reiterate / re‑research / verify) → ready‑to‑work → in‑progress → done
  • Cron worker: background poller claims ready‑to‑work tasks + a changes watcher
  • Human‑in‑the‑loop input requests: agent pauses → quick‑choice chips + custom input → resumes
  • Full auth: accounts, login, sessions, hashed passwords
  • Teams: create, join, roles, per‑team task visibility
  • MCP interface for agents (token‑authed)
  • Buildless web board: Kanban + detail panel + live agent status
  • Global, project‑keyed storage (survives repo moves/reinstalls)
  • Ported card contract: root‑cause / solution / diagram callouts

→ Deferred (v2+)

  • Public managed hosting & multi‑instance ops
  • Phone‑optimized responsive UI + Cloudflare Tunnel remote
  • Playwright / Chromium deterministic verification
  • Persistent Chrome‑profile test runs
  • Background merge/verify status tracker — toggleable, cron/agent‑run, keeps done·verified·merged live
  • Agent‑driven generative UI — the agent picks what to render for input/approval (schema · HTML · iframe · buttons · MCP action), not a fixed widget
  • GitHub PR integration (v2) — link a task to its PR, pull PR review comments/threads in, and resolve or reply from be10x with two‑way sync back to GitHub
  • External task‑tracker sync (do‑it, Jira, …)
  • Task‑type marketplace / sharing
  • Cross‑team analytics & fleet orchestration
  • Postgres migration (only if hosted at scale)
