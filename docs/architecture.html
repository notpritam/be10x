be10x · architecture

How the whole thing works

One rule holds the design together: sessions are disposable, state is durable. The board is the single source of truth; the agent that does the work is a throwaway process that can run on any machine — yours, a teammate's, or a server — and reports everything back to the board.

01

Three moving parts

Everything in be10x is one of three things. Keeping them separate is what lets the same system run on your laptop today and across a team tomorrow — only the wire between them changes.

02

Today vs. the team & remote model

You asked whether the flow is “a hosted part + a part running on their system.” Yes — that's exactly it. The only difference between what runs today and the team version is the wire between the local agent and the board.

The honest status: the local model (left) is built and running. The team/remote model (right) is the same code with one change — the gfa_* MCP and runner speak to the board over HTTP instead of opening a local SQLite file. Nothing else in the design has to move.
A shareable URL today, without waiting on that: put the host behind a tunnel — Cloudflare Tunnel, ngrok, an app like Tunnels — pointed at be10x serve's port. No code change: it's the same single-host board from the left panel, just reachable over a real https:// link instead of localhost. That covers "thin teammates in the browser" today; distributed runners and a multi-writer hosted board are still the planned work above.
03

How one step actually runs

Nothing is “always on.” A human action on the board drops a wake into a queue; a runner claims it, runs exactly one step, and the loop comes back to you. This is why a server restart or a closed laptop never loses work — the queue and the state are durable.

04

Plans & the task lifecycle

A task walks a fixed set of states. Plans and findings are stored on the task as HTML (rendered safely in a sandbox), so they're readable by everyone with access — not trapped in someone's terminal. Two states are yours to hold (teal): the plan waits for your approval, and a question waits for your answer.

1 Backlog→ 2 Researching→ 3 Plan review ✋→ 4 Ready→ 5 In progress→ 6 Verifying→ 7 Done
↳ Needs input ✋ (pauses mid-work for your answer) ↳ Blocked
  • Plan review is the control point: the agent proposes, you approve or comment. A comment doesn't just chat — it drops a revise wake and the plan gets reworked.
  • Artifacts (RCA, diagrams, findings, verification) hang off the task as HTML cards, so the “why” travels with the work and shows up for reviewers too.
  • Verify is a deliberately fresh session that only sees the approved plan + the code — it checks the work honestly instead of trusting its own build transcript.
05

Teams & shared access

Everyone authenticates against the one board. A team groups people and tasks; a role decides what they can do; a task's scope decides who can see it. Add teammates by searching the platform or picking recent collaborators.

  • Scopes: personal (just you) · project (a linked repo) · team (everyone in the team).
  • Share links let someone without an account review a task through a token URL — read the plan & artifacts, comment, leave a verdict, and (if the link permits) ask the agent to pick it up. This is your “review it to go live” path.
  • Access without a login stays read-limited: a share link is comment_only or run_agent — never full board access.
06

Running commands on their system

This is the part that surprises people, so it's worth stating plainly: the agent runs wherever the runner that claimed the wake is running. The board never reaches into anyone's machine — a machine opts in by running a runner and pulling work down.

  • Pull, not push. A teammate's runner asks the board for the next wake for its projects and claims it. The board can't execute anything on a machine that isn't running a runner.
  • Where the commands land. Whoever's runner claims the job runs the shell/edits/tests in their own clone, inside a fresh git worktree per task — so parallel tasks never collide, and the checkout is the safety boundary.
  • Two boundaries. Auth = a personal token + team membership (who may claim what). Safety = the per-task worktree the agent is confined to, where it runs unattended.
  • Crash-safe. If a runner dies mid-step, the board detects the dead process and can recover the task — the durable state and the queue survive; only the throwaway session is lost.
So, whose computer does the work? The one whose runner took the job. Put a runner only on your host and everything runs there (thin teammates in the browser). Put a runner on each teammate's laptop and each person's work runs locally in their own repo. Put a runner on a VM and that's your remote session. Same board, your choice per machine.
07

Remote sessions & “adopt to the board”

A runner on a server or VM is a remote session — it claims wakes and works exactly like a local one, and you steer it entirely from the board. The reverse direction exists too: pull an ad-hoc terminal session onto the board.

  • Adopt (built): working in any terminal that has be10x linked, say “move this to the 10x board” (or run /be10x-adopt / be10x adopt). One MCP call files it as a task in the right project, at the phase it's really in — carrying whatever exists: title, summary, plan, findings-as-artifacts, refs.
  • Then the board owns it. Optionally hand off, and a runner (yours or a teammate's or a VM's) continues the work; otherwise you drive it from the board.
  • The “key” you asked for is exactly this: one trigger that lifts loose terminal work into durable, shared, controllable board state — and from then on you control it from the board.
